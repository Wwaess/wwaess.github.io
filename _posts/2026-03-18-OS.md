# Operating System Security

An **operating system** is a set of programs that manage a computer system's behaviour and how other programs interact witch each other on this system. These provide _interfaces_ for this interaction. 

## Interfaces 

USER interfaces include _graphical_ (GUI) and _text-based_ (TUI); the latter is often called the _command line interface_, or *CLI*. 

Other interfaces include *application programming interfaces (API)* and *application binary interfaces (ABI)*. 

Lastly, _device drivers_ are interfaces that connect physical hardware. 

OSes handle resource management (i.e. of the CPU, memory and files) via *threads*. 

User interface design is so important that an entire fields - such as HCI (human computer interaction) and UX (user experience [aesthetics]) - have spawned as a result. 

Most resorce management, which includes: 
- memory management
- process management
- energy management
- communication management
- file management

is handled from a supervisory module called a **kernel**. Though most OS models combine these, there are 3 core "classical" models for a kernel:
- monolithic
- micro-kernel
- exo-kernel. 

Strictly, a **process** is a block of memory containing executable code, process-specific data, a stack for subroutine data, a heap for runtime data (e.g. calcualtions), pointers (reference markers) to external data (other files, etc.), security attributes (think IDs), processor states and register contents. 

**Threads** are to scheduling what processes are to resource allocation. A process must have at least one thread, which are tracked by the OS to schedule the process's execution. To avoid CPU cycle wastage, the OS provides a scheduler to keep the CPU busy when it's idle - which is going to be a lot of the time. 

## Virtualisation/Abstraction
These are key attributes of most modern OSes. 

## Services and Security
A **service** is a catch-all term for any functionality that the OS provides to support hard-/software or user tasks. A **print service** (involving printer drivers, tools to manage the printer, and spooling/caching capabilities for print jobs) is a common example. 

# History

1. Early OSes were simple batch systems (e.g. LEO in 1951, for repetitive business tasks like payrolls and order processing). 
2. The Atlas Supervisor was developed in the late '50s and introduced interrupt-driven operation, a hardware- and application-managing supervising program, and time-sharing between programs. At the time, security was only concerned insofar as _physical access_ was pretty much the only way you could access a computer. 
3. In the turn of th '60s, the _Compatible Time Sharing System (CTSS)_ was pioneered which could run batch programs in the background (i.e. what "compatible" means in this case). 
4. The *IBM System 360 (S/360)* introduced *Complex Instruction Set Computer (CISC)* concepts and early forms of virtualisation. 
5. Early _malware_ formed, notably the CHRISTMA Worm, written in REXX and started on IBM mainframes. 
6. Unix was created in 1969 after Bell Labs left Multics, aiming for a simpler, lightweight time‑sharing system.
7. Unix spread widely because it was portable (rewritten in C), small, and easy for universities to adopt and modify.
8. Early Unix had weak security, with issues in permissions, software quality, and inconsistent third‑party add‑ons.
9. Thompson’s Trusting Trust showed that a compiler can hide backdoors even if the OS source code looks clean.
10. GNU and later Linux rebuilt Unix‑like tools openly, with modern hardening added through projects like SELinux and AppArmor.
11. CP/M introduced modular microcomputer OS design (BIOS/BDOS/CCP) and heavily influenced MS‑DOS.
12. MS‑DOS originated from QDOS, a CP/M‑like clone Microsoft licensed and adapted for IBM PCs.
13. Floppy‑disk boot sector viruses (starting with Elk Cloner in 1981) became the first widespread microcomputer malware.
14. GUIs (from Engelbart’s 1968 demo to 1980s systems like MacOS, AmigaOS, GEM/TOS) massively expanded attack surfaces.
15. Early‑2000s worms (Code Red, Nimda, SQL Slammer) exploited Microsoft server flaws, prompting the Trustworthy Computing Initiative.

# Saltzer and Schroeder Principles
1. **Economy of Mechanism** - simple design vs. optimised no. of system features
2. **Least Privilege** - that a user can perform what they need to do with the least amount of privilege needed
3. **Open Design** - "not security through obscurity" (Kerckhoff / Petitcolas 1997)
4. **Fail-Safe Defaults** - _deny as a default_
5. **Separation of Privilege** - e.g. 2FA
6. **The Least Common Mechanism** - assisted or segregated resources shoudl avoid being shared? 
7. **Complete Mediation** - permissions and access control is regularly checked
8. **Psychological Acceptability** - security measures should not increase cognitive load
9. **Work Factor** - e.g. longer passwords and 256-bit keys [NOTE - how does that match the name of this principle!?]
10. **Compromise Recordings** - auditing and logging

# Secure Programming Languages, Protocols and Methodologies

Secure languages aim to eliminate common vulnerability classes such as buffer overflows, injection flaws and information leaks. PL/I in Multics is an early example of such a language. 

Safe abstractions, such as managed memory and taint-tracking as found in Perl, Ruby and PHP, and such as Python's "faceted values" extention, are easy ways to manage potential data leakage. 

Modern languages like Rust and Go provide built-in memory and thread safeties. 

# Security Extensions

Because legacy code often has to be kept for compatibility reasons, security flaws are often _patched_ via extensions, rather than rewriting the system entirely. 

SE Linux aims to extend the Linux kernel with security enhancements and mandatory access control (MAC). 

AppArmor provides simpler MAC via per‑app profiles with limited permission flags (r, w, m, x).

Antivirus tools evolved from simple virus scanners into full security suites (firewall, spam, VPN, etc.).

Unix‑like systems rely less on AV due to built‑in firewalls, signed software ecosystems, and restrictive default permissions.

Proxy servers filter traffic, scan mail, and enforce separation of privilege without altering OS design.
